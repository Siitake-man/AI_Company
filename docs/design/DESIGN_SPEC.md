# AI Team Builder（AIカンパニー）アプリケーション設計書

**バージョン:** 2.0（Manus/Genspark引き継ぎ後、Claude壁打ちによる最終FIX版）

---

## 1. 概要とコンセプト

### 1.1 プロダクト概要
- プロダクト名（仮）：**AIカンパニー**
- コンセプト：「あなただけのAI専門家チームを持とう」
- NotebookLMで26人の"雇われAI社員"と対話した体験からインスピレーションを得て、その体験をローカルデスクトップアプリとして再構築する。ただし本アプリの本質は**組織の再現**ではなく、**一人の運営者に専門知識を後付けで届けること**にある。

### 1.2 本質的な位置付け（重要な前提）
- 本アプリは、実在する人間やチームを仮想空間に置き換えるものではない。
- 想定シーンは「一人（または少人数）で開発・運営しているプロジェクトに対し、社内に相談できる同僚や上司がいない状態」。
- そこに、法務・マーケティング・エンジニアリングなど各分野の専門知識を持つAI社員を"雇う"ことで、専門性を後付けし、意思決定の材料を増やす。
- 比喩でいえば、**一人の身体に複数の専門家の頭脳を後付けする装置**であり、組織図のコピーではない。

### 1.3 主な使い方（2ユースケース）
- **(a) 個人プロジェクトの意思決定支援**：個人で運営する実プロジェクト（例：NPO-Trust）において、専門知識を持つAI社員に相談・壁打ちしながら、意思決定に必要な事実・PRO/CON・視点を集める。最終的な結論は運営者本人が下す（原則4：ユーザーが結論を出す、と対応）。
- **(b) 社内向けの議論トレーニング**：SES企業などの社内メンバーが、AI社員同士の会議シミュレーションを通じて、議論の進め方や多角的な視点の持ち方を学ぶ教材として活用する。

> ※(a)(b)いずれも、実在の人間・チームの代替や再現を目的としたものではなく、運営者一人を支える"仮想の専門家チーム"という位置付けである。

### 1.4 プロジェクトの独立性
- 1つのAIカンパニーアプリ内に、複数の独立したプロジェクトを作成できる。
- 各プロジェクトは専用のチーム（部署・メンバー）を持ち、プロジェクト間でメンバーは共有されない（同じ肩書でも所属プロジェクトが違えば人格も別）。

## 2. コアプリンシプル（絶対に変えない原則）

**原則1: BYOK（Bring Your Own Key）。** ユーザーが自身のAPIキー（OpenAI / Anthropic / Google Gemini）を入力して利用する。APIキーはサーバーに送信されず、中央管理も行わない。各ユーザーのデバイス内にのみ保管される。

**原則2: ローカル完結。** プロジェクト情報・チーム構成・会話ログ・議事録は、すべてユーザーのPC内のローカルSQLiteに保存される。外部への送信は、LLM APIへのリクエストを除いて一切行わない。

**原則3: 使うほど賢くなる（自動学習フィードバックループ）。** ユーザーの価値観・判断理由の蓄積により、使い続けるほどAIメンバーの回答精度・文脈適合度が向上する設計とする。具体的には**方向性②（決定事項からの自動学習）**を採用する。ユーザーが会議（S8）で下した「決定事項（Decisions）」が、その会議の参加メンバーの学習データに自動的に注入され、次回以降の会議や1on1でその決定ルールを前提として行動する。手動での過度なプロンプト修正を不要とし、ツールを使えば使うほど自然にAI社員が組織の文化と決定を覚えていく「成長日誌」の仕組みを実装する。

**原則4: 結論はユーザーが出す。** AIメンバーは、断定的な結論や意思決定そのものをユーザーに代わって下す存在ではない。事実・PRO/CON・見解を提示することで、ユーザーが自ら判断するための材料を揃える役割を担う。この原則は特に「探索モード」の会議において厳格に適用され、AIが勝手に結論を先取りしないよう、システムプロンプト・議事録フォーマットの両面で徹底する。

## 3. データ構造（3階層構造・4層マージ）

本アプリの根幹をなす構造であり、DBスキーマ・全画面UIの両方に影響する。

```
AIカンパニー(アプリ)
└── プロジェクト(会社)   ← 例: NPO-Trust
    └── 部署(部門)       ← 例: 法務部 / 戦略部 / エンジニア部
        └── メンバー(個人) ← 例: 契約担当 / セキュリティエンジニア / ドリーマー
```

**プロジェクト**は、名称・目的・ユーザーの判断軸・価値観を持つ。この「判断軸・価値観」は、そのプロジェクトに属する全ての部署・メンバーのシステムプロンプトに自動的に注入される層である。

**部署**は、単なるメンバーの整理棚ではなく、独自の「部署としての性質・行動の癖」を持つ人格継承型のハコとして実装する。例えば法務部には「慎重でリスク回避的、契約書の一言一句にこだわる」といった部署プロンプトを持たせ、その部署に属する全メンバーのシステムプロンプトに中間層として注入する。これにより、新しいメンバーを部署に追加した際、ゼロから人格プロンプトを書かずとも、その部署らしい思考の癖を自動的に受け継ぐ。

**メンバー**は、名前・役割・専門領域・人格プロンプト・アバター・利用AIモデルを持つ、最も内側の層である。

**システムプロンプトの継承ロジック（4層マージ）**は、以下の順序で構築される。

```
最終システムプロンプト =
  ユーザーのコアプロフィール(最外層・全プロジェクト共通・S9で設定)
  + プロジェクトの価値観・判断軸(S3で設定)
  + 部署の性質・行動の癖(S4/S5で設定)
  + 個人の役割・専門領域・人格プロンプト(S5で設定)
```

### 部署プリセット

ユーザーはプロジェクト作成時にプリセット部署を選択でき、カスタムでの追加・削除・名称変更も可能とする。各プリセット部署には、初期状態で部署プロンプト（らしさ）を用意し、ユーザーが自由に編集できるようにする。

| 部署 | 部署としての性質（初期プロンプトの方向性） | デフォルトメンバー例 |
|---|---|---|
| 経営・戦略 | 大局観を持ち、優先順位とリソース配分を重視する | 経営戦略担当、相談役、PM |
| エンジニアリング | 実現可能性と保守性を重視し、技術的リスクに敏感である | UI/UXデザイナー、セキュリティエンジニア、インフラエンジニア |
| 法務・コンプライアンス | 慎重でリスク回避的、契約書・規約の一言一句にこだわる | 契約担当、コンプライアンス担当、知財担当 |
| マーケティング | 機会とスピードを重視し、市場・顧客視点で発想する | 戦略担当、SNS担当、データ分析担当 |
| 思考スタイル | 部署としての性質は持たず、個人ごとに独立した思考の型を代弁する | ドリーマー、悪魔の代弁者、現実路線、ファシリテーター |

### 思考スタイル型メンバー

通常の専門家メンバーとは異なり、「思考の多様性」を担保するための特殊メンバーである。会議ごとにON/OFFを切り替えられる。これらは部署の性質を継承せず、プロジェクト固有の人格として個別に機能する。プロジェクトをまたいで共有しない。

| スタイル | 役割 |
|---|---|
| ドリーマー | 夢と可能性を論じる。プロジェクトへの情熱を代弁する |
| 悪魔の代弁者 | リスクを顕在化する。「なぜ失敗するか」を論じる |
| 現実路線 | 実行可能性の番人。コストと制約の中で考える |
| ファシリテーター | 議論を統合し、論点を整理する（結論を断定はしない） |

## 4. 技術スタック

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | React + TypeScript + Tailwind CSS | 型安全性が高く、AIエージェントによる実装・レビューがしやすい |
| デスクトップ化 | Tauri（Rust） | Electronと比較して軽量、セキュリティモデルが強固、社内配布時の実行ファイルサイズ・配布コストが低い |
| データストア | SQLite（ローカル） | 3階層構造や会話ログ・議事録といった構造化データを、外部DBサーバー不要でローカルに永続化できる |
| AIオーケストレーション | Phase 1: 自前実装（素朴なfor文によるラウンドロビン制御） / Phase 2以降: LangChain.jsへの部分的な差し替えを検討 | Phase 1では学習コストと抽象化レイヤーを避け、動作の切り分けをしやすくする。Phase 2でワークフロー型会議など柔軟なフロー制御が必要になった段階で導入し、ユーザーの学習体験として位置づける |
| RAG（知識ベース・価値観蓄積） | Phase 1: 導入しない / Phase 2: ベクトルDB導入（技術選定は未定） | Phase 1はスコープを絞る。Phase 2導入時は「AIエージェント構築における検索拡張生成の仕組みそのものを学習する機会」として位置づける |
| APIキー保管 | Tauriのsecure storage | macOSはKeychain、WindowsはDPAPI、LinuxはSecret Serviceを利用し、平文でのディスク書き込みを行わない |

**補足（AIオーケストレーション）:** Phase 1の実装は「メンバーを順番に呼び出す」という処理を、将来LangChain.jsに差し替えても外側のコードに影響が出ないよう、薄いインターフェース（契約）として切り出しておく。これにより「動かない原因が自分のロジックかフレームワークか」を切り分けやすくする。

**補足（RAG）:** Phase 2でRAGを導入する際は、知識ベースはプロジェクト単位で管理する（同じ倉庫、違うレンズ方式）。検索クエリには「問いかけたメンバーの役割コンテキスト」を含めることで、同じ知識ベースから、メンバーの役割に応じた異なる検索結果を引き出す設計とする。知識をメンバーごとに分離するのではなく、アクセスは共有しつつ、役割というフィルターを通して自然に異なる情報が優先される仕組みを目指す。

## 6. 画面設計

※ UIのビジュアルデザインやコンポーネントルール（レトロRPG・ピクセルアート風）の詳細は、[DESIGN_SYSTEM.md](./DESIGN_SYSTEM.md) を参照のこと。

### 6.1 画面遷移

```
S1(初回起動・APIキー設定)
    ↓ APIキー登録完了
S2(プロジェクト一覧 / ホーム)
    ├── 新規作成 → S3(プロジェクト作成)→ S4(チーム管理)
    ├── プロジェクト選択 → S4(チーム管理)
    └── 設定アイコン → S9(設定)

S4(チーム管理)
    ├── 部署タブ切り替え(プロジェクト内)
    ├── メンバー追加/編集 → S5(メンバーエディタ)→ S4
    ├── 「話す」ボタン → S6(1on1チャット)→ S4
    └── 「会議を開始」ボタン → 会議モード選択(探索/収束) → S7(会議モード)→ S8(議事録)→ S4
```

### 6.2 画面定義（簡潔）

**S1（初回起動・APIキー設定）:** プロバイダー選択・APIキー入力・ローカル保存確認。Web検索（NotebookLM機能）のためのTavily / Braveキー設定も可能。未設定の場合はOpenAI/Gemini等の組み込み検索機能で代替するフォールバック設計とする。

**S2（プロジェクト一覧）:** 複数プロジェクトの一覧表示・メンバープレビュー・会議クイックスタート。

**S3（プロジェクト作成）:** プロジェクト名・目的・価値観の入力、部署プリセットの選択（カスタム追加可）、初期メンバー選択。部署プリセットを選ぶ際、各部署の「らしさ（部署プロンプト）」の初期値がプレビュー表示され、その場で編集できる。

**S6（1on1チャット）:** 特定メンバーとの直接対話・履歴表示。

**S9（設定）:** APIキー管理・デフォルトモデル・アプリ設定・データ管理に加え、**コアプロフィール（シンプル版）**を追加。テキストエリア一枚で「あなたについて、AI社員が知っておくべきことを自由に記述してください」という入力欄を設ける。この内容は4層マージの最外層として全プロジェクト・全メンバー共通で注入される。他AIからの情報抽出プロンプト付きUIなど凝った機能はPhase 2で拡張する。

### 6.3 画面定義（詳細）

#### S4: チーム管理

**主な機能:** プロジェクト内の部署タブ切り替え、部署ごとのメンバー一覧表示、部署の人格継承バッジ表示、メンバー追加/編集、1on1開始、会議開始。

画面上部に現在選択中のプロジェクト名を表示し、その下に部署タブ（経営・戦略／エンジニアリング／法務／マーケティング／思考スタイル、等）を横並びで配置する。各部署タブを開くと、その部署の人格継承バッジ（「NPO-Trustの価値観を継承中」等）が上部に小さく表示され、その下に該当部署に属するメンバーのカードが一覧表示される。

#### S5: メンバーエディタ

**主な機能:** アバター選択、役割・専門領域・人格プロンプト・AIモデル設定、および**成長日誌（自動学習履歴）の確認・編集**。

画面上部に「継承元」を示す小さなパネルを設け、現在編集中のメンバーがどのプロジェクト・どの部署に属しているかを表示し、それぞれの価値観・部署プロンプトを折りたたみ式で確認できるようにする（例:「▶ NPO-Trustの価値観を見る」「▶ 法務部の性質を見る」）。
さらに、そのメンバーが過去の会議の決定事項から自動学習した履歴（成長日誌）をタイムライン形式で表示する。ユーザーは自動学習された内容の確認、手動での追記、不要になった古いルールの削除を行える。

#### S7: 会議モード

**主な機能:** マルチエージェント自律会議（ラウンドロビン方式）、議題管理、割り込み機能、一時停止機能、発言中メンバーのハイライト。

**会議開始前のモード選択:** S4から「会議を開始」ボタンを押した直後、会議モードを選択するステップを挟む。**探索モード**（材料を集め、結論はユーザーが出す会議）と**収束モード**（ファシリテーターが最終ラウンドで推奨案の提示まで導く会議）の2択とし、会議開始時に固定する（会議中の切り替えはPhase 2で対応）。

**発言と割り込みの状態機械:** 会議中の状態は「発言中」「割り込み受付中」「一時停止中」の3つを行き来する。あるメンバーが発言を終えると、10秒間の「割り込み受付ウィンドウ」が視覚的に強調表示される。ただし割り込みは常時可能であり、10秒経過後もボタンは有効なまま残る。割り込みボタンを押すと、割り込まれた対象メンバーがまずユーザーの発言内容を踏まえて追加発言を行い、その後ラウンドロビンの順番は次のメンバーに進む。次のメンバーは、元の議題・直前の発言・ユーザーの割り込み内容・割り込み後の追加発言、これら全てをコンテキストとして受け取る。同一メンバーへの連鎖的な割り込みは最大3回までとし、上限に達した時点で割り込みボタンをグレーアウトする。

**一時停止ボタン:** 押した瞬間、現在表示中の発言はそのまま表示を継続するが、次のメンバーへのバトンタッチ（次の発言生成リクエスト）を止める。割り込み受付ウィンドウのカウントダウンが動いていた場合は、その時点で凍結し、再開ボタンで残り時間から再スタートする。この一時停止状態はアプリのメモリ上でのみ管理し、DBには保存しない（アプリ再起動でリセットされて構わない）。

#### S8: 議事録・サマリー

**主な機能:** 決定事項・ネクストアクション・追加論点の表示、Markdownエクスポート、**決定事項のDB保存に伴うAIメンバーへの自動学習トリガー**。会議モードによって出力内容が分岐する。

**探索モードから遷移した場合:** 「論点」「各メンバーの見解（PRO/CON形式）」「事実ベースの材料」を主役とした構成とし、決定事項欄は空欄のテンプレートとして提示し、ユーザー自身が記入する。

**収束モードから遷移した場合:** 上記に加えて画面最上部に「AI社員一同の提言」という専用セクションを新設する。参加メンバー全員のアバターを横並びに配置し、アンバー色の太い縁取りで囲んだ特別な枠として表示する。この枠内には、ファシリテーターがまとめた推奨案、各メンバーのこの提言への納得度（星やゲージ等）、特定メンバーが依然懸念を残している場合はその一言メモを添える。

**自動学習（方向性②）のトリガー:** ユーザーが「決定事項」を入力して「議事録を保存」ボタンを押した際、その会議の参加メンバーの学習データ（SQLiteの `member_learnings` テーブル）に、その決定内容と日付が自動的に INSERT される。次回以降のシステムプロンプト生成（4層マージ）の際に、この学習履歴が動的にマージされる。

**Markdownエクスポート:** 探索モード用テンプレートと収束モード用テンプレート of 2種類を用意し、会議モードに応じて自動的に出力形式を切り替える。

### 7.3 生成ロジックの補足

議事録生成時にLLMへ渡すプロンプトには、「決定事項を断定的に記入してはならず、探索モードでは空欄のテンプレートとして出力すること」「収束モードにおいても、AI社員一同の提言はあくまで参考情報であり、最終決定事項の欄はユーザーの記入待ちとして空欄にすること」という指示を明記する。これは原則4（結論はユーザーが出す）を、議事録生成のプロンプトレベルでも徹底するための措置である。

## 8. セキュリティ設計

APIキーはTauriのsecure storage（macOS: Keychain、Windows: DPAPI、Linux: Secret Service）に保存し、平文でのディスク書き込みは行わない。プロジェクトデータ・部署データ・メンバーデータ・会話ログ・議事録・ユーザーのコアプロフィールは、すべてローカルSQLiteに保存し、外部への送信は行わない（LLM APIへのリクエストを除く）。

会議ログ・1on1ログには、ユーザーの割り込み発言やコアプロフィールの内容が含まれるため、これらのデータもローカル保存の対象として同列に扱い、外部送信の例外扱いは設けない。LLM APIへのリクエスト時にのみ、必要なコンテキスト（システムプロンプト・会話履歴の一部）が各プロバイダー（OpenAI / Anthropic / Google）に送信されることをS1（初回起動時）の同意画面で明示する。

## 9. 開発フェーズ

### Phase 1（今月中・約38時間・社内発表用MVP）

3階層構造（プロジェクト・部署・メンバー）とその4層マージによるシステムプロンプト継承ロジック、部署の人格継承（人格継承型）、S1〜S9の全画面実装、コアプロフィール（シンプル版・S9に追加）、会議モード（ラウンドロビン方式・探索/収束の2モード制）、割り込み機能（常時可能・10秒強調ウィンドウ・連鎖最大3回・一時停止ボタン）、議事録生成（探索/収束それぞれのテンプレート・Markdownエクスポート）、APIキー設定（OpenAI / Anthropic / Gemini）。

AIオーケストレーションは自前のシンプルな実装（ラウンドロビンのfor文制御）とし、LangChain.jsは導入しない。RAG（ベクトルDB）も導入せず、会議ログはSQLiteへの単純な蓄積・直近履歴の再利用に留める。

### Phase 2（来月以降・優先順位順）

1. 知識ベース + Fast Research（RAG導入。知識ベースはプロジェクト単位で管理し、検索クエリにメンバーの役割コンテキストを含める「同じ倉庫、違うレンズ」方式を採用する。ベクトルDBの技術選定を行い、学習体験として位置づける）
2. コアプロフィールの本格作り込み（他AIからの情報抽出プロンプト付きUI、Phase1のシンプル版からの拡張）
3. 価値観の蓄積・学習機能 + 従業員成長ログ（マーケティング系メンバーとセキュリティ系メンバーで、成長ログの指標を分ける。前者は判断理由の蓄積件数、後者は最終知識更新日・参照した情報の鮮度）
4. ワークフロー型会議（企業の意思決定プロセス模擬。LangChain.jsへの部分的な差し替えを実施し、学習体験として位置づける）
5. 会議モードの途中切り替え（探索⇔収束）
6. 非同期会議モード
7. ローカルLLM接続（LM Studio / Ollama）
8. MCPサーバー化（Cursor等から`ask_member`等で呼び出し）
9. Obsidian MCP接続
10. テンプレートギャラリー

### Phase 3（将来構想）

クラウド同期（APIキーは各デバイスのみ）、モバイルアプリ（iOS / Android）、サブスク（月額150〜200円 = クラウド同期利用料）。

### 設計上の注意点・破綻リスク

| 項目 | リスク・注意事項 |
