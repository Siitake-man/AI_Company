## 習得した技術・概念

### 1. Tauriのセキュアストレージ（keyring）

- **What**: OSが提供する「パスワード金庫」にデータを保存するTauriの機能。
- **Why**: APIキーをSQLiteに平文保存すると、DBファイルを盗まれたとき即座に漏洩する。keyringを使えばOSが暗号化して管理するため、ファイルを盗んでも復号できない。
- **Where**: `src/lib/apiKeyStore.ts` で実装。`saveApiKey`, `getApiKey`, `deleteApiKey` の3関数でラップしている。
- **身近な例え**: Windowsの「資格情報マネージャー」やMacの「キーチェーン」に保存するのと同じ仕組み。アプリが「金庫の番号を持って金庫番に渡す」だけで、平文は自分のポケットに入れない。

### 2. Tauri v2 のパーミッション設定

- **What**: `src-tauri/capabilities/default.json` に、Rustのコマンドを呼び出す許可を明示的に書く設定ファイル。
- **Why**: Tauri v2はゼロトラスト原則を採用しており、何もしなければフロントエンドはバックエンドのRust関数を一切呼べない。呼びたい操作（DB読み書き、keyring保存等）を1つずつ許可する必要がある。
- **Where**: `src-tauri/capabilities/default.json` 内の `"permissions"` 配列に `"allow-select"`, `"allow-execute"` 等を追記。
- **身近な例え**: 会社のビルに入館証を発行するイメージ。Rustの各機能は「部屋」で、フロントエンドは「来訪者」。`capabilities` は「どの来訪者がどの部屋に入れるか」を書いた入館ルール台帳。

### 3. SQLiteの `INSERT OR REPLACE` のカスケードトラップ

- **What**: `INSERT OR REPLACE` を使ったとき、主キー衝突があると内部的に「古いレコードをDELETEしてからINSERT」する。
- **Why（問題）**: `ON DELETE CASCADE` が設定されたテーブルがあると、この内部DELETEが連鎖し、子テーブルのデータが全消滅する。
- **対策**: 初期データ投入時は `INSERT OR REPLACE` を使わず、「データが0件の場合のみ `INSERT` を実行する」ロジックに変更した。子から親の逆順でDELETEしてからINSERTするのも有効。
- **身近な例え**: 「社員登録名簿で名前が重複したら古い行ごと削除して入れ直す」命令が、「その社員に紐づく契約書も全部シュレッダーにかけてしまう」状態。重要データがある場合は更新（UPDATE）か、存在チェック後にINSERTを使うべき。

### 4. 4層マージシステムプロンプトのロジック

- **What**: AI社員1人が受け取るシステムプロンプトを、4つの情報源から合成する仕組み。
- **構成層**（外側から順番に）:
  1. ユーザーのコアプロフィール（全AI社員共通の文脈）
  2. プロジェクトの価値観・目的（このプロジェクト専用の文脈）
  3. 部署の性質・行動の癖（法務部なら「慎重・リスク回避」等）
  4. メンバー個人の人格プロンプト（最も具体的なキャラクター定義）
- **例外**: 部署が `is_thinking_style=1`（思考スタイル型）の場合、第3層（部署性質）を**スキップ**する。ドリーマーや悪魔の代弁者は「部署の色」に染まらず、純粋な思考の型を代弁するため。
- **実装場所**: `src/lib/promptMerger.ts` の `getMergedSystemPrompt()` 関数。

### 5. DESIGN_SYSTEM.md（デザインシステム文書化）

- **What**: モックアップから抽出したカラーコード・フォント・CSSクラス定義を1ファイルに集約したドキュメント。「唯一の正（Single Source of Truth）」として機能する。
- **Why**: 複数のAIが画面を実装するとき、ルールが1箇所にまとまっていないと毎回「一般的なモダンSaaSデザイン」が出力されてしまう。DESIGN_SYSTEM.mdを読ませることで、どのAIが担当しても「レトロRPG・ドット絵」の世界観を維持できる。
- **Where**: `docs/design/DESIGN_SYSTEM.md`。全AI実装者（Antigravity/Jules）への強制ルールとして §6 に明記。
- **身近な例え**: 飲食チェーン店の「レシピ本兼接客マニュアル」。どの店舗・どのスタッフが対応しても、同じ味・同じ接客品質になるための統一基準書。

### 6. AGENTS.md（AIゲートウェイファイル）

- **What**: プロジェクトルートに置くことで、AntigravityやJulesが作業開始前に**自動的に最初に読む**特殊なファイル。
- **Why**: 設計書を読む順番・絶対NG事項・優先順位をここに書くことで、「誰も教えてないAIが誤った方向に実装する」リスクをゼロにする。
- **Where**: `C:\Users\bonob\Project\AI_Comany\AGENTS.md`（プロジェクトルート）。
- **詳細版ルール**: `docs/design/AI_RULES.md` がAGENTS.mdの詳細版として機能する関係。

## 遭遇したエラーと解決策

| 事象 | 原因 | 対策 |
|---|---|---|
| `INSERT OR REPLACE` で子テーブルのデータが全消滅 | `ON DELETE CASCADE` + 内部DELETE起動のカスケード | `memberCount === 0` のときのみ `INSERT` を実行するロジックに変更 |
| Tauriのフロントエンドからsqlやkeyring機能が呼べない | Tauri v2のゼロトラスト原則により、全機能がデフォルト禁止 | `src-tauri/capabilities/default.json` に `allow-select`, `allow-execute`, keyring関連パーミッションを個別に追記 |
| フロントエンド変更とバックエンド変更の反映タイミングが異なる | Reactはホットリロード対応だが、Rust（src-tauri）変更は再コンパイルが必要 | `npm run tauri dev` を停止→再起動して完全再コンパイルを実施 |
| DESIGN_SPECとDESIGN_SYSTEMでカラーコードが不一致 | 設計書を複数箇所に分散して書いたため矛盾が発生 | DESIGN_SYSTEM.mdをSSoTとして確定し、DESIGN_SPEC §5.1 を「廃止→参照」に切り替え |
