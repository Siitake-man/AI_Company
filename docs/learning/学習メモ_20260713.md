## 2. 習得した技術と概念（NotebookLM抽出用詳細）

### 2.1 SQLiteの外部キー制約と「INSERT OR REPLACE」の罠
*   **What（何をするものか）**:
    *   `ON DELETE CASCADE`は、親データ（例：部署）が削除された際、それに紐づく子データ（例：その部署に所属するメンバー）も自動的に削除して、データの整合性（親のいない迷子データを作らない）を保つ機能。
    *   `INSERT OR REPLACE` は、主キーが重複したレコードがあれば自動で上書きしてくれる便利なSQLコマンド。
*   **Why（なぜトラブルになったか / 事象と原因）**:
    *   SQLiteの仕様上、`INSERT OR REPLACE`は内部的に「既存のレコードを一度完全に削除（DELETE）してから、新しいレコードを挿入（INSERT）する」という挙動をする。
    *   そのため、アプリ起動時に親である「部署」データを上書き（REPLACE）しようとした瞬間、内部でDELETEが走り、`ON DELETE CASCADE` 設定によって「部署に紐づいていた全メンバーの子データが芋づる式に自動消去される」という事態が発生した。
*   **身近な例えでの理解**:
    *   オフィスの「部署の看板（親）」を新しいものに掛け替える作業を頼んだつもりが、業者のマニュアル（仕様）では「一度その部署の部屋ごと一度解体して（CASCADE）、新しい部屋を作る」となっており、結果として中にいた「社員（子）」が全員いなくなってしまった、という状態。
*   **対策とベストプラクティス**:
    *   初期化時は `REPLACE` を使わず、手動で「子（メンバー）→親（部署）→大元（プロジェクト）」という依存の末端から順番に安全に `DELETE` を行い、その後に通常の `INSERT` を行うロジックに変更。また、「すでにデータが1件以上あれば初期化処理自体をスキップする」ことで、毎回削除・挿入の負荷とリスクを避ける設計にした。

### 2.2 OSのセキュアストレージとTauri (Rust) の連携
*   **What（何をするものか）**:
    *   ユーザーのAPIキー（OpenAIなど）を、アプリ独自のデータベースやテキストファイルではなく、OS（WindowsやMac）自体が提供している厳重な資格情報管理システムに保管する技術。
    *   今回はRustの`keyring`クレート（ライブラリ）を採用し、Tauriのカスタムコマンドとして実装。
*   **Why（なぜ必要か）**:
    *   APIキーは「会社の銀行印やクレジットカード」と同じ。平文（暗号化されていないテキスト）でパソコンの中に置いておくと、他の悪意あるアプリやウイルスに簡単に盗まれてしまう。
    *   OSのセキュアストレージは、他のアプリが勝手に覗き見できないようにOSレベルで暗号化・保護されているため、ローカル完結型アプリにおける最高のセキュリティ担保手段となる。
*   **Where/When（どこで使うか）**:
    *   パスワード、APIトークン、クラウド連携の認証情報など、「絶対に外部に漏れてはいけないデバイス固有の機密情報」を保存する際に必ず使用する。
*   **身近な例えでの理解**:
    *   暗証番号をメモして机に貼り付けておく（ファイル保存）のではなく、**「OSという非常に警備の厳しい銀行の貸金庫」**を借りてそこに預けるイメージ。アプリが鍵を使いたいときは、銀行員（OS）に都度リクエストを出して安全に取り出す。

### 2.3 Tauri v2の権限管理（Capabilities）とホットリロードの境界
*   **What（何をするものか）**:
    *   Tauri v2では、フロントエンド（React/JavaScript）側が、OSの機能（ファイル操作やSQLデータベース実行）を勝手に呼び出せないよう、`capabilities/default.json` というファイルで明示的に「許可証」を発行するゼロトラスト（何も信じない）設計になっている。
*   **Why（なぜエラーになったか / 事象と原因）**:
    *   SQLの実行権限としてざっくり`sql:default`と記述していましたが、v2の最新仕様では「読み込み(`allow-load`)」「書き込み(`allow-execute`)」「検索(`allow-select`)」と権限が細分化されており、明示的に記述しないとセキュリティブロック（アクセス拒否）が働いてエラーになった。
*   **開発サイクルの気付き（ホットリロードの落とし穴）**:
    *   React側（画面の見た目や文字）の修正は、保存した瞬間に即座に画面に反映される（ホットリロード）。
    *   しかし、Rust側（`Cargo.toml`へのライブラリ追加や、`capabilities`の設定変更、Tauriコマンドの追加）は、「アプリの根幹となる設計図」の変更であるため、**一度アプリを完全に終了して、コンパイル（ビルド）をし直さないと反映されない。** 

### 2.4 4層マージロジックの概念と「思考スタイル」のオーバーライド
*   **What（何をするものか）**:
    *   AI社員の人格を、「1. ユーザーのコアバリュー」「2. プロジェクトの目的」「3. 部署の性質」「4. 個人の役割」の4つの階層からなるプロンプトを合成して生成するロジック。
*   **Why（なぜ必要か）**:
    *   単なる「優秀なAI」ではなく、「しいたけさんの価値観」と「プロジェクトのゴール」を共有したうえで、自律的に判断できる真のチームメンバーを作るため。
*   **特定の例外処理（思考スタイル）**:
    *   「悪魔の代弁者」など、特定の思考フレームワーク自体を役割とするメンバーの場合、部署の「慎重」「積極的」といった性質（3層目）を混ぜると、本来の純粋な批判的思考がブレてしまう。
    *   そのため、「思考スタイル型の部署・メンバーの場合は、3層目のマージをスキップし、1+2+4層だけでプロンプトを構成する」という分岐ロジックを実装し、テスト画面でその動きが完璧に機能することを実証した。

### 2.5 DESIGN_SYSTEM.md (UIデザインの唯一の正 / SSoT)
*   **What（何をするものか）**:
    *   モックアップのピクセルアート（ドット絵）風デザインを元に、カラーコードやフォント、立体感のあるボタンなどのCSSコンポーネント実装のルールをドキュメント化したもの。
*   **Why（なぜ必要か）**:
    *   複数のAIエージェント（Antigravity/Jules）が並行・非同期でUIを実装する際、統一のカラーやフォントの基準がないと、各自がバラバラなTailwindクラスを適用して「一般的なのっぺりしたモダンWebデザイン」に戻ってしまい、手戻り（生命の浪費）が発生するため。
*   **身近な例えでの理解**:
    *   テーマパーク（ディズニー等）のデザインガイドライン。アトラクションや看板、ゴミ箱に至るまで、共通の「世界観」を守るための配色ルールブック。

### 2.6 AGENTS.md (AIゲートウェイファイル)
*   **What（何をするものか）**:
    *   プロジェクトのルートディレクトリに配置し、AIエージェント（AntigravityやJules）が作業開始前に必ず自動で一番最初に読み込むように指定された行動規範ファイル。
*   **Why（なぜ必要か）**:
    *   AIに「絶対に触ってはいけない3層構造・4層マージのコアロジック」や「デザインシステム順拠ルール」を最優先で認識させ、独断での簡略化や見落としを完全に防ぐため。
*   **身近な例えでの理解**:
    *   工場の作業場に入る前に全員が必ず唱和する「安全第一の3箇条」看板。

## 3. 遭遇したエラーと解決策

| 事象 | 原因 | 対策 |
|---|---|---|
| `INSERT OR REPLACE` で子テーブルのデータが全消滅 | `ON DELETE CASCADE` + 内部DELETE起動のカスケード | `memberCount === 0` のときのみ `INSERT` を実行するロジックに変更 |
| Tauriのフロントエンドからsqlやkeyring機能が呼べない | Tauri v2のゼロトラスト原則により、全機能がデフォルト禁止 | `src-tauri/capabilities/default.json` に `allow-select`, `allow-execute`, keyring関連パーミッションを個別に追記 |
| フロントエンド変更とバックエンド変更の反映タイミングが異なる | Reactはホットリロード対応だが、Rust（src-tauri）変更は再コンパイルが必要 | `npm run tauri dev` を停止→再起動して完全再コンパイルを実施 |
| DESIGN_SPECとDESIGN_SYSTEMでカラーコードが不一致 | 設計書を複数箇所に分散して書いたため矛盾が発生 | DESIGN_SYSTEM.mdをSSoTとして確定し、DESIGN_SPEC §5.1 を「廃止→参照」に切り替え |
