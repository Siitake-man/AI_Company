# 学習メモ — 2026-07-18（AIカンパニー開発）

## サマリー（10〜15行）
本日は設定画面のUI刷新（2ペイン構成への変更およびタブの統合）と、ボタン操作が機能しないバグの修正を行いました。特に「接続テストが動かない」「Markdownで保存しようとするとエラーが出る」という現象は、Reactレイヤーのバグではなく、**Tauri v2の強力なセキュリティパーミッション**が原因であることが判明しました。
Tauri v2では、単にファイルシステムへのアクセスを許可（`fs:default`）するだけでなく、書き込み操作自体（`fs:allow-write-text-file`）や、特定の対象ディレクトリに対する権限（`fs:allow-home-write-recursive` など）を明示的にホワイトリスト登録しなければならないという学びを得ました。
また、任意のプロジェクトで「AIカンパニー用のコンテキスト情報」を自律的に収集・出力するためのIDE向けSKILLを構築し、グローバル領域へ配置しました。これにより、各プロジェクトの目的や技術スタックの把握が容易になり、AIカンパニー内のエージェントが高度な文脈を持って議論に参加できるようになります。

## 習得した技術・概念
- **What（何をするものか）**: Tauri v2のCapabilities（パーミッションシステム）
- **Why（なぜ必要か）**: デスクトップアプリがOSのファイルシステムやネイティブAPI（ダイアログ等）へ勝手にアクセスして悪用されるのを防ぐため、最小権限の原則（PoLP）に基づき、必要な権限だけを明示的に付与する。
- **Where（どこで使うか）**: `src-tauri/capabilities/default.json` や各種設定ファイル内。
- **身近な例えでの理解**: 
  > 会社のセキュリティゲートのようなもの。「入館証（fs:default）」を持っているだけでは「金庫室（特定のフォルダ）」や「シュレッダーの操作（ファイルの書き込み）」はできない。ゲートを通過するだけでなく、特定の部屋と具体的な操作ごとの「追加の許可証（fs:allow-write-text-file等）」を事前申請して首から下げておかないと、システムに弾かれてしまう。

## 遭遇したエラーと解決策
| 事象 | 原因 | 対策 |
|---|---|---|
| 設定画面の「接続テスト」や、サマリー画面の「Markdownで保存」ボタンを押しても何も起こらない、または `fs:write_text_file not allowed` というエラーダイアログが出る。 | Tauri v2のパーミッション設定（`capabilities/default.json`）において、`dialog:default`、`fs:default` に加え、具体的な書き込み権限（`fs:allow-write-text-file` 等）が欠落していたため、OSネイティブへのブリッジアクセスが拒否された。 | `default.json` の `permissions` 配列に、`dialog:default`, `fs:default`, `fs:allow-write-text-file`, `fs:allow-home-write-recursive` 等の権限を明記し、アプリを再ビルドした。 |
| グローバルSKILLを作成したが、ローカルの古いSKILLをターミナルから `rm` コマンドで自動削除できなかった。 | PowerShellやWindows環境特有のアクセス権限（ACLエラー等）またはプロセスのロックにより、ファイル削除コマンドが拒否された。 | 無理にコマンドで削除せず、古いファイルの中身を「グローバルへ移動した」という案内テキストに上書きし、ユーザーに手動削除を促した。 |
