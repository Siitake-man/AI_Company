# スキル：セッション開始準備 (session-start) — AIカンパニー専用

このスキルは、「AIカンパニー」プロジェクトの新しい開発セッションを開始する際に、AIが自律的に実行する準備・同期手順です。
前回の文脈とJulesの成果を迅速に復元し、5分以内に作業を開始できる状態を作ります。

---

## プロジェクト固有パス定義

| 変数名 | 実際のパス |
|---|---|
| `TaskFile` | `docs/design/ROADMAP.md` |
| `PlanFile` | `docs/design/DESIGN_SPEC.md` |
| `SpecsDir` | `docs/design/` |
| `LearningMemoDir` | `docs/learning/` |
| `HandoverDir` | `docs/jules-tasks/` |
| `SchemaFile` | `docs/design/DATA_SCHEMA.md` |
| `RulesFile` | `docs/design/AI_RULES.md` |

## 実行フロー

AIは本スキルを検知またはユーザーから「セッション開始」「session-start」「再開します」などと指示された場合、以下のステップを**すべて自律的に一括で実行**します。

### ステップ1：最新コンテキストの自動スキャンと同期

AIはプロジェクト内の以下のファイルを順番に読み込み、現在のプロジェクト状況を完全に復元します。

1. **`docs/design/ROADMAP.md`**: ROADMAPのノードA〜Hの完了状態を確認し、現在地を特定する
2. **`docs/design/DESIGN_SPEC.md`**: データ構造・画面定義・4層マージロジックの最新仕様を確認する
3. **`docs/design/DATA_SCHEMA.md`**: 現在のSQLiteスキーマ状態を確認する
4. **`docs/design/AI_RULES.md`**: 実装上の制約・禁止事項を再確認する
5. **`docs/learning/` 内の最新の学習メモ**: 前回の振り返りと「次回の最初のアクション」を確認する

### ステップ2：Julesの成果確認と同期（非同期リレーの受け取り）

AIは、Julesが夜間に進めていた作業を確認します。

1. **Julesの引き継ぎメモの読み込み**:
   - `docs/jules-tasks/jules_handover_YYYYMMDD.md`（存在する場合）を読み込み、完了タスクと残課題を把握する
   - `docs/learning/学習メモ_Jules_YYYYMMDD.md`（存在する場合）を確認し、技術的な変更を把握する

2. **用済み指示書のアーカイブ**:
   - `docs/jules-tasks/prompt_for_jules_YYYYMMDD.md` が存在し、対応するJulesの作業が完了していれば、`docs/jules-tasks/archive/` へ移動させてクリーンアップする

3. **コード変更の確認（git）**:
   - しいたけさんに「`git log --oneline -5` でJulesの最新コミットを確認してください」と提案する（自動実行はしない）

### ステップ3：環境の健全性チェック（AIカンパニー固有）

AIは以下をチェックし、問題があれば報告します。

1. **プロジェクト構造の確認**:
   - `src/`（Reactフロントエンド）と `src-tauri/`（Rustバックエンド）の主要ファイルが存在するか確認する
   - `src-tauri/migrations/`（SQLiteマイグレーションファイル）が存在するか確認する（ROADMAPノードA完了後）

2. **設計書との整合性確認**:
   - `docs/design/` 配下の設計書に「TODO」「要確認」「暫定」などのフラグが残っていないかチェックする

3. **インデックス確認**:
   - `Codebase-memory-mcp` のインデックス更新が必要かどうかを確認する（前回コード変更があった場合）

### ステップ4：5分マイクロタスクの特定

AIは以下の情報を統合し、今日の最初の「Small Win」を定義します。

- 前回の学習メモの「次回再開時の最初のアクション」
- Julesの成果による完了タスクの差し引き
- ROADMAPの次の未完了ノード

**特定の優先ルール（AIカンパニー固有）:**
- ROADMAPのノードA（DBスキーマ）が未完了 → 最優先
- ノードAが完了、ノードB未完了 → 4層マージロジックの単体テスト
- ノードB完了後 → S1→S9→S2→S3→S4→S5→S6→会議モード選択→S7→S8→Markdownエクスポートの順

### ステップ5：ユーザーへのセッション再開宣言

すべての準備が完了した後、AIはしいたけさんに以下のフォーマットで報告し、合意を得てから最初のタスクに着手します。

```markdown
### 🌅 セッション再開準備完了（AIカンパニー）

前回の文脈およびJulesの作業成果を同期・復元しました。

**【現在の状況】**
- **ROADMAPの現在地**: ノード○（[ノード名]）完了 → ノード○（[ノード名]）着手中
- **Julesによる変更**: ○○（なければ「Julesの作業なし」）
- **設計上の注意点**: ○○（なければ省略）
- **未解決の課題**: ○○（なければ省略）

**【技術スタックのリマインド】**
React + TypeScript + Tailwind CSS / Tauri (Rust) / SQLite / 自前ラウンドロビン制御

**【今日の最初のアクション（5分以内）】**
- [ ] ROADMAPノード○: [具体的な作業内容を1行で]
  > ヒント: [実際のファイル名や実行すべきコマンド]

それらの確認から作業を開始してよろしいでしょうか？
```

---

## 留意事項

- ローカル開発サーバー（`npm run dev` / `cargo tauri dev`）の起動は自動で行わず、ユーザーへの確認後に提案する
- Julesの引き継ぎファイルのアーカイブは、タスク完了を確認してから実行する（誤操作防止）
- `Codebase-memory-mcp` の更新が必要な場合は、ユーザーに確認してから `index_repository` を実行する
- 3層構造・4層マージに影響する変更がJulesにより行われていた場合、必ずしいたけさんに確認を求める（AI_RULES.md §8）
