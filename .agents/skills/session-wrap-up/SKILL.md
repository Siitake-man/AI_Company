# スキル：セッション終了振り返り (session-wrap-up) — AIカンパニー専用

このスキルは、「AIカンパニー」プロジェクトの開発セッション終了時に、AIが自律的に実行する振り返り・引き継ぎ手順です。
しいたけさんの可処分時間を最大化し、「再開時にゼロから思い出す時間」を完全に排除します。

---

## プロジェクト固有パス定義

| 変数名 | 実際のパス |
|---|---|
| `TaskFile` | `docs/design/ROADMAP.md` |
| `PlanFile` | `docs/design/DESIGN_SPEC.md` |
| `SpecsDir` | `docs/design/` |
| `LearningMemoDir` | `docs/learning/` |
| `HandoverDir` | `docs/jules-tasks/` |

---

## 実行フロー

AIは本スキルを検知またはユーザーから「セッション終了」「wrap-up」などと指示された場合、追加の対話による時間浪費を避けるため、以下のステップを**すべて自律的に一括で実行**します。

### ステップ1：セッションファクトの構築（脳内整理）

AIはまず、本セッションで実行した以下の情報を整理します。

- `CompletedTasks`: ROADMAPのどのノードが完了・進行中か（A〜Hのノード表記で記載）
- `Troubles`: 遭遇したエラー・原因・対策の3点セット（Tauri/Rust/React/SQLiteそれぞれ区別して記録）
- `ModifiedFiles`: 作成・修正したファイルと、4層マージ構造における役割（どの層に影響するか）
- `SpecsModified`: `docs/design/`内で変更が必要になった設計書
- `JulesTasks`: Jules（夜間エージェント）に委ねるべきタスク（主にCSS装飾・リファクタリング・軽微なバグ修正）
- `NextAction`: 次回再開時に迷わず5分で着手できる超具体的なファーストアクション（ROADMAPのノード名を明記）

### ステップ2：既存ドキュメントの自動更新

AIは以下のドキュメントを読み込み、自律的に更新します。

1. **`docs/design/ROADMAP.md` の更新**:
   - 完了したノードに `✅ YYYY-MM-DD完了` を追記する
   - 「2. ノードの意味と完了条件」の該当行の完了条件を更新する

2. **`docs/design/DESIGN_SPEC.md` の更新**:
   - データ構造・画面定義に変更があった場合のみ、該当セクションを更新する
   - 変更がない場合はスキップして構わない

3. **`docs/design/DATA_SCHEMA.md` の更新**:
   - SQLiteスキーマに変更があった場合のみ更新する（3層構造・4層マージの根幹のため慎重に）

> ⚠️ `AI_RULES.md §8`：3層構造・4層マージに影響する変更はユーザー承認が必要。自律更新の対象外とし、確認を促すこと。

### ステップ3：不要スペックの自動アーカイブ

`docs/design/` 内をスキャンし、最新仕様と矛盾・重複するドキュメントが存在する場合は、
削除せず `docs/design/archive/` へ移動させてクリーンアップします（フォルダが存在しない場合は自動作成）。

### ステップ4：学習メモの自動生成

以下のパスに詳細な学習メモを新規作成します。

- **保存先**: `docs/learning/学習メモ_YYYYMMDD.md`

**内容の要件（NotebookLM連携を想定・極力詳細に）:**

```markdown
# 学習メモ — YYYY-MM-DD（AIカンパニー開発）

## サマリー（10〜15行）
[今日の作業全体の概要]

## 習得した技術・概念
- **What（何をするものか）**: 
- **Why（なぜ必要か）**: 
- **Where（どこで使うか）**: 
- **身近な例えでの理解**: 
  > [電話帳・工場のベルトコンベア等の比喩]

## 遭遇したエラーと解決策
| 事象 | 原因 | 対策 |
|---|---|---|

## AIカンパニー固有の設計判断
- ROADMAPの現在地（ノードX完了 → ノードY着手中）
- 4層マージへの影響（変更がある場合）
- 3層構造（プロジェクト→部署→メンバー）への影響（変更がある場合）

## 次回再開時の最初のアクション（5分で着手できるレベル）
- ROADMAPノード: [ノード名]
- 具体的な作業: [ファイル名と操作内容を1行で]
```

### ステップ5：Jules向け引き継ぎ指示書の自動生成

以下のパスにJules向けの指示書を作成します。

- **保存先**: `docs/jules-tasks/prompt_for_jules_YYYYMMDD.md`
- ファイル名ルール（`AI_RULES.md §6`）: 同日複数タスクの場合は `_01`, `_02` の連番付与

**内容の要件:**

```markdown
# Jules向け指示書 — YYYY-MM-DD（AIカンパニー）

## 対象ブランチ・コミット
- ブランチ: [現在のブランチ名]
- 直前のコミット: [コミットハッシュ / メッセージ]

## 依頼するタスク
[具体的なファイルパスと作業内容を日本語で詳述]

## 作業上の制約（必ず守ること）
- Tauri / React / TypeScript / Tailwind CSS / SQLite のスタック以外は使用しない
- `src-tauri/` 配下の Rust コードは触らない（フロントエンドのみ）
- 3層構造（プロジェクト→部署→メンバー）と4層マージロジックは変更しない
- ハードコード禁止（デザイントークン・プリセットはconstファイルで管理）

## 設計資料の参照先
- DESIGN_SPEC.md: `docs/design/DESIGN_SPEC.md`
- DATA_SCHEMA.md: `docs/design/DATA_SCHEMA.md`
- AI_RULES.md: `docs/design/AI_RULES.md`

## 完了条件と確認方法
[Julesが作業完了と判断できる具体的な基準]

## 注意・申し送り
[AntigravityからJulesへの特記事項]
```

### ステップ6：ユーザーへの最終出力（3大打返し）

すべての生成・更新が完了した後、以下をユーザーに提示します。

1. **【今日やったこと＆詰まったポイント】** ROADMAPノードの進捗と、事象・原因・対策の3点セット。

2. **【Git一括コマンド】** コピー＆ペーストだけでコミット＆プッシュが完了するコマンド:
   ```powershell
   git add . && git commit -m "feat: [ノード名] YYYY-MM-DD" && git push
   ```

3. **【Jules Web UI用指示プロンプト】** JulesのUI上にそのままコピペできる短いテキスト:
   > 「`docs/jules-tasks/prompt_for_jules_YYYYMMDD.md` を読み込んで対応してください。」

4. **【次回再開時の魔法のプロンプト】**:
   > 「AIカンパニー開発を再開します。`docs/design/` 配下の設計書と `docs/learning/学習メモ_YYYYMMDD.md` を読み込み、ROADMAPの現在地を確認してから、前回の続きの作業を提案してください。」

---

## 留意事項

- `git` コマンドや `npm` コマンドの自動実行は行わず、ステップ6でユーザーに提示して実行を促す（AI_RULES.md準拠）
- 3層構造・4層マージに影響する変更は、自律更新の対象外としてユーザーへ確認を求める
- Tauri固有ファイル（`src-tauri/`配下）の変更は特に慎重に記録し、Julesへの依頼対象から除外する
